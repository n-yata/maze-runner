# コミット & Push

**引数:** コミットメッセージのヒント（省略可。例: `/commit-push フルーツボーナス実装`）

---

## ステップ1: 変更内容の確認

以下を並行して実行する:

1. `git status` — 未ステージ・ステージ済みファイルを確認
2. `git diff HEAD` — 変更内容を確認
3. `git log --oneline -5` — 直近のコミット履歴とメッセージスタイルを確認

コミット対象のファイルを特定し、シャビに変更内容を簡潔に報告する。

---

## ステップ2: セキュリティレビュー (クルトワ)

CLAUDE.md のルールに従い、コミット前にクルトワ（security-engineer エージェント）にレビューを依頼する。

- **model**: `opus`
- **レビュー対象**: ステップ1で特定した変更ファイルすべて
- **必須チェック項目**:
  - XSS・インジェクション (OWASP A03)
  - URL・シークレット・API キーのハードコーディング
  - 認証・認可の不備 (OWASP A01, A07)
  - テストコードに意味のないアサーションがないか

**Critical / High の指摘があった場合**: 修正してから再度このステップを実行する。

---

## ステップ3: コミットメッセージの作成

変更内容から適切なコミットメッセージを作成する:

- 形式: `<type>: <説明>`
- type: `feat`（新機能）/ `fix`（バグ修正）/ `refactor`（リファクタリング）/ `test`（テスト）/ `docs`（ドキュメント）/ `chore`（その他）
- 引数でヒントが与えられた場合はそれを参考にする
- 日本語で簡潔に（体言止め推奨）

---

## ステップ4: コミット & Push

1. 変更ファイルをステージング（機密ファイル `.env` 等を除く）
2. コミット（HEREDOC 形式で Co-Authored-By を付与）
3. `git push` でリモートに push
4. 完了をシャビに報告（コミットハッシュと push 結果を含む）
